## The decision

- The original "pipeline" Phase 6 says: *train a real baseline anomaly scorer on the flow/log dataset.*
- The "Improvements" Phase 6 says: *hand-build a synthetic event log — "not a real anomaly detector."*

These are opposite calls. We must pick one, together, now.

## Recommendation: TWO ENGINES — don't choose, but don't merge

### Engine 1 — Real detection on real data *(Technical Excellence, 20%)*
Anomaly / lateral-movement detection trained and evaluated on **CICIDS-2017** and **LANL**, scored against LANL's red-team ground truth.

**Why we can't drop this:** we already hold 11 GB of verified data, and LANL gives us what almost no team will have — **749 labeled red-team events across 104 compromised users (days 1–29 of 58)**, real attacker lateral movement in real enterprise logs. Published results on this exact dataset (e.g., ~85% TPR at 0.9% FPR for graph-based detectors; RAID/USENIX and later work) give us a credible target and a citable benchmark context. If our whole demo runs on a hand-written CSV, Technical Excellence collapses — it's the current `demo_ps7_pipeline.py` problem at larger scale.

*(Fun fact confirming feasibility: the demo scenario Codex generated already borrowed LANL's real red-team source host `C17693` — the bridge from mock to real data is natural.)*

### Engine 2 — ATT&CK next-technique prediction + attribution *(Innovation, 25%)*
The teammates' new centerpiece (per the DeepOP-style papers). Keep it — it's the rare, memorable claim: *"we predict the attacker's next 2–3 moves and rank likely APT groups."*

**Demo narrative that joins them:** Engine 1 flags real anomalies → correlation builds the chain → Engine 2 predicts next techniques + attributes the actor → SOAR recommends gated containment.

## Corrections to the Tab-2 plan (keep the plan, fix these three)

### 1. Sequence count: 15–25 is too small — and unnecessary. Use ~150–200.
Measured from our own `enterprise-attack.json`:

| Metric | Value |
|---|---|
| Active APT groups | 174 |
| Campaigns (with `uses` relations) | 55 |
| Groups with ≥6 techniques | **145** |
| Campaigns with ≥6 techniques | **51** |
| **Total auto-buildable sequences** | **~196** |
| Mean techniques per group sequence | 30.8 |
| Prediction vocabulary (distinct techniques used) | 525 |

An LSTM/Transformer trained on ~15 sequences learns nothing; on ~196 it's borderline viable — **scale Phase 2 to the full pool** (fully scriptable from the STIX file we already have). Keep the 3–5 hand-curated CERT-In sequences as the manual differentiator.

### 2. The circularity trap (the one that can sink us in Q&A)
"Improvements" Phase 2 orders each group's techniques by **ATT&CK kill-chain tactic order**, then Phase 4 trains a model to predict the next technique. If the labels are generated by an ordering rule, the model mostly re-learns that rule — an ML-literate judge will see it. Also measured: **21% of techniques span multiple tactics**, so the ordering isn't even well-defined for 1-in-5 steps.

**Fixes (do all three):**
- Predict **within** the imposed order what the rule cannot know: *which specific technique comes next* (the rule only fixes the tactic ordering, not the technique choice among 525).
- Always report lift **over two baselines**: (a) most-frequent-next-technique, (b) kill-chain-order heuristic itself. If a first-order **Markov transition model** matches the neural model, present the Markov model — honest and defensible (there is published work on ATT&CK technique co-occurrence supporting this framing).
- Evaluate manual (CERT-In) vs heuristic sequences separately, as the Improvements doc already suggests. State the split openly in the pitch.

### 3. Vocabulary vs data reality
525-technique vocabulary with ~196 sequences is thin. Mitigations: predict **top-k (k=3/5)** as planned; optionally back off to **tactic-level prediction** (14 classes) as the headline metric with technique-level as secondary. Sentence-transformer embeddings of technique descriptions (as planned) help generalization — good call by the team.

## What the literature says (grounding, not homework)

- **DeepOP** (the MDPI paper the team cited — the "causal window attention" figure in our doc) is exactly this task: transformer over ATT&CK sequences extracted from CTI. We are *reproducing its shape at hackathon scale* — legitimate, but their dataset needed author contact; our auto-built sequence pool replaces it.
- **KillChainGraph** (arXiv 2508.18230) does phase-aware technique prediction with ATTACK-BERT embeddings — same embedding trick as our Phase 3; validates the approach.
- **Technique co-occurrence** (arXiv 2211.06495) supports Markov/association baselines — our honesty baseline.
- **TRAM** (MITRE CTID) — use as planned in Phase 2 for auto-suggesting technique tags from CERT-In advisories; don't build our own NLP extractor.
- **LANL lateral-movement literature** (GL-GV, LMDetect, USENIX RAID '20) — Engine 1 has citable benchmarks; ~85% TPR @ <1% FPR is a realistic bar to reference.
