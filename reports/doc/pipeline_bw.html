The analysis pipeline: seven stages, one function call

Every stage runs live on whatever log is submitted. Nothing is pre-computed at request time.
1
Feature engineering
Turn each raw login record into seven behavioural measurements that describe how this account normally moves through the network.

Anomaly scoring
An IsolationForest trained only on normal behaviour scores every event from 0 to 100. Scores use fixed calibration anchors, so a score means the same thing on every upload.

3
Correlation
Every event scoring 50 or above becomes an alert. Alerts are grouped into a single incident with a timeline, a severity and an ordered chain of techniques, instead of thousands of separate rows.
src/shared/correlate.py · ALERT_THRESHOLD = 50 · SESSION_GAP = 3600 s
SHARED SPINE
|
4
MITRE ATT&CK mapping
Each behaviour is matched to an official technique identifier. Names and descriptions are read from the parsed MITRE bundles, never generated, so an invented technique is structurally impossible.
src/shared/attack_mapper.py · 918 techniques
SHARED SPINE
|
5
Attack path graph
Alerts become edges in a directed graph of machine to machine movement. From it we compute total exposure, which valuable machines are reachable, and which single machine to isolate first.
src/shared/attack_graph.py · networkx · reachability from every pivot
SHARED SPINE
|
6
Gated response (SOAR)
Containment steps are drawn from the real MITRE mitigations for the observed techniques. Anything touching a critical asset requires human approval. All actions are simulated and labelled as such.
src/shared/soar.py
SHARED SPINE
|
7
Prediction and attribution
A first order Markov model ranks the attacker's likely next technique with a real transition probability. A transparent weighted retrieval ranks which known threat group the behaviour resembles, with a written justification.
src/engine2/build_predictor.py · src/engine2/attribution.py
ENGINE 2
Predict and attribute
One bundle out. All seven stages are a single function, analyze_events(), which returns one result object containing the payload for every screen in the application. The sample data shipped with the app is produced by calling the same function on a shipped log, so there is no separate demonstration path that could diverge from the real one.
